# UI Design

## 1. 개요

Spring Boot + Thymeleaf 웹 어드민 시스템의 UI/UX 디자인 원칙을 정의한다. 명확성·존중·깊이를 핵심 테마로 하며, 웹 브라우저 환경에 맞게 적용한다.

**핵심 테마:**

| 테마 | 설명 |
|------|------|
| **명확성** | 텍스트는 읽기 쉽고, 아이콘은 정확하며, 장식은 기능에 종속된다 |
| **존중** | 콘텐츠가 주인공이다. UI 요소는 콘텐츠를 돕되 경쟁하지 않는다 |
| **깊이** | 시각적 레이어와 모션이 계층 구조와 활력을 전달한다 |

**적용 범위:**

| 포함 | 제외 |
|------|------|
| 디자인 원칙, 레이아웃, 타이포그래피 | 특정 플랫폼 API·SDK |
| 색상, 아이콘, 컴포넌트, 네비게이션 | 특정 프레임워크 구현 코드 |
| 인터랙션, 접근성, 안티패턴 | 프로젝트 고유 디자인 시스템 |

## 2. 디자인 원칙

### 2.1 5대 원칙

| 원칙 | 원문 | 핵심 |
|------|------|------|
| **일관성** | Consistency | 동일한 동작에 동일한 UI를 사용한다 |
| **직접 조작** | Direct Manipulation | 화면 요소를 직접 조작하는 느낌을 제공한다 |
| **피드백** | Feedback | 모든 사용자 행동에 즉각 반응한다 |
| **메타포** | Metaphors | 현실 세계의 경험을 디지털에 반영한다 |
| **사용자 제어** | User Control | 사용자가 항상 주도권을 갖는다 |

### 2.2 일관성 (Consistency)

동일한 기능은 앱 전체에서 동일한 외형과 동작을 가진다. 플랫폼이 제공하는 표준 컴포넌트와 패턴을 우선 사용한다.

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 모든 화면에서 동일한 버튼 스타일 사용 | 화면마다 다른 버튼 형태 |
| 플랫폼 표준 네비게이션 패턴 적용 | 독자적 네비게이션 패턴 발명 |
| 동일 아이콘은 동일 의미 전달 | 같은 아이콘을 다른 기능에 재사용 |

### 2.3 직접 조작 (Direct Manipulation)

콘텐츠를 직접 터치·드래그하여 조작하는 느낌을 제공한다. 중간 단계를 최소화하고 결과를 즉시 보여준다.

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 사진을 핀치로 확대/축소 | 확대 버튼 → 배율 선택 → 적용 |
| 리스트 항목을 드래그하여 재정렬 | 편집 모드 → 순서 번호 입력 → 저장 |

### 2.4 피드백 (Feedback)

모든 사용자 액션에 시각·청각·촉각 피드백을 제공한다.

| 상황 | 필수 피드백 |
|------|------------|
| 버튼 탭 | 하이라이트 상태 변화 |
| 폼 제출 성공/실패 | 성공 메시지 또는 에러 필드 하이라이트 + 메시지 |
| 네트워크 요청 중 | 로딩 인디케이터 표시 |
| 삭제 동작 | 확인 대화상자 → 완료 피드백 |

### 2.5 메타포 (Metaphors)

현실 세계의 경험에서 차용한 시각적·동작적 단서를 활용한다. 사용자가 학습 없이 직관적으로 이해할 수 있도록 한다.

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 휴지통 아이콘으로 삭제 표현 | 추상적 기호로 삭제 표현 |
| 토글 스위치로 ON/OFF 표현 | 텍스트 링크로 상태 전환 |

### 2.6 사용자 제어 (User Control)

사용자가 항상 현재 상태를 이해하고, 이전 단계로 되돌아갈 수 있도록 한다. 되돌릴 수 없는 동작 전에 반드시 확인을 요청한다.

| 규칙 | 설명 |
|------|------|
| **실행 취소 지원** | 가능한 모든 동작에 Undo를 제공한다 |
| **뒤로 가기 보장** | 모든 화면에서 이전 화면으로 복귀할 수 있다 |
| **파괴적 동작 확인** | 삭제·초기화 등은 확인 대화상자를 거친다 |
| **강제 중단 방지** | 작업 도중 강제로 다른 흐름에 진입시키지 않는다 |

## 3. 레이아웃

### 3.1 8pt 그리드 시스템

모든 간격과 크기를 **8pt 단위**로 설정한다. 미세 조정이 필요하면 4pt 단위를 허용한다.

| 단위 | 값 | 용도 |
|------|-----|------|
| 기본 단위 | 8pt | 여백, 패딩, 컴포넌트 크기 |
| 미세 단위 | 4pt | 아이콘 내부 여백, 텍스트 간격 |

> 8의 배수(8, 16, 24, 32, 40, 48...)를 우선 사용한다. 홀수 픽셀 값은 사용하지 않는다.

### 3.2 여백 체계

| 단계 | 값 | 용도 |
|------|-----|------|
| **Compact** | 8pt | 관련 요소 간 밀착 간격 |
| **Regular** | 16pt | 기본 컴포넌트 간 간격 |
| **Generous** | 24pt | 섹션 간 구분 |
| **Spacious** | 32pt 이상 | 주요 영역 간 구분 |

- 여백 단계를 일관되게 적용한다. 화면마다 임의 값을 사용하지 않는다.
- 콘텐츠 좌우 마진은 **Regular(16pt)** 이상으로 확보한다.

### 3.3 Safe Area

고정 헤더·사이드바·툴바가 콘텐츠를 가리지 않도록 레이아웃을 구성한다.

| 규칙 | 설명 |
|------|------|
| 고정 UI 아래 콘텐츠 | 고정 헤더·사이드바 높이만큼 콘텐츠 영역에 offset을 적용한다 |
| 스크롤 콘텐츠 | 고정 요소와 겹치지 않도록 padding 또는 margin으로 여백을 확보한다 |

### 3.4 반응형 기준점

CSS 미디어 쿼리 기준으로 3단계 브레이크포인트를 사용한다.

| 구간 | 기준 | 레이아웃 전략 |
|------|------|-------------|
| **Mobile** | ~768px 미만 | 사이드바 숨김, 햄버거 메뉴 |
| **Tablet** | 768px~1280px | 사이드바 축소 또는 선택적 표시 |
| **Desktop** | 1280px 이상 | 사이드바(220px) 상시 표시, 다중 컬럼 |

- 이 프로젝트의 주 대상은 **Desktop**이다. Mobile·Tablet 대응 범위는 별도 정의한다.
- 각 구간에서 콘텐츠가 자연스럽게 배치되는지 확인한다.

### 3.5 정렬 원칙

| 규칙 | 설명 |
|------|------|
| 텍스트 기본 정렬 | **좌측 정렬** (LTR 언어 기준) |
| 숫자·금액 | **우측 정렬** — 자릿수 비교가 용이하다 |
| 제목·헤더 | 좌측 정렬 또는 중앙 정렬을 일관되게 적용한다 |
| 아이콘 + 텍스트 | 아이콘과 텍스트의 수직 중앙을 맞춘다 |
| 폼 레이블 | 입력 필드와 동일한 좌측 기준선에 정렬한다 |

## 4. 타이포그래피

### 4.1 서체 역할 체계

역할 기반 타이포그래피 체계를 사용한다. 절대 크기보다 **역할(role)** 로 서체를 지정한다.

| 역할 | 용도 | 참고 크기 | 무게 |
|------|------|----------|------|
| **Large Title** | 최상위 화면 제목 | 34pt | Bold |
| **Title 1** | 주요 섹션 제목 | 28pt | Bold |
| **Title 2** | 보조 섹션 제목 | 22pt | Bold |
| **Title 3** | 소제목 | 20pt | Semi-Bold |
| **Headline** | 강조 텍스트 | 17pt | Semi-Bold |
| **Body** | 본문 텍스트 | 17pt | Regular |
| **Callout** | 보조 설명 | 16pt | Regular |
| **Subhead** | 리스트 보조 텍스트 | 15pt | Regular |
| **Footnote** | 각주, 부가 정보 | 13pt | Regular |
| **Caption 1** | 캡션 | 12pt | Regular |
| **Caption 2** | 보조 캡션 | 11pt | Regular |

> 참고 크기는 기본값이다. 브라우저 텍스트 줌 적용 시 크기가 조정된다 — §10.2절 참고.

### 4.2 가독성 규칙

| 규칙 | 기준 |
|------|------|
| **최소 본문 크기** | 11pt 이하로 내려가지 않는다 |
| **행간(Line Height)** | 글자 크기의 **1.2~1.5배** |
| **문단 최대 폭** | 한 줄에 **40~60자(한글)** 또는 **60~80자(영문)** |
| **자간(Letter Spacing)** | 시스템 기본값을 사용한다. 임의 조정하지 않는다 |
| **서체 수 제한** | 하나의 화면에 **최대 2개** 서체 패밀리를 사용한다 |

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| Body 역할로 본문 지정 | 15px 고정 크기로 본문 지정 |
| 시스템 기본 서체 사용 | 장식 서체를 본문에 사용 |
| 계층 구조에 따라 크기·무게 조합 | 모든 텍스트에 동일 크기 적용 |

## 5. 색상

### 5.1 시맨틱 컬러

용도에 따라 의미를 가진 색상 토큰을 정의하고 사용한다. 하드코딩된 색상값을 직접 사용하지 않는다.

| 토큰 | 용도 | 설명 |
|------|------|------|
| **Primary** | 주요 액션, 브랜드 강조 | 앱의 대표 색상 |
| **Secondary** | 보조 액션, 부가 요소 | Primary와 구분되는 보조 색상 |
| **Destructive** | 삭제, 경고 액션 | 빨간 계열 — 위험한 동작을 나타낸다 |
| **Success** | 성공 상태 | 녹색 계열 — 완료·성공을 나타낸다 |
| **Warning** | 주의 상태 | 노란/주황 계열 — 주의를 나타낸다 |
| **Background** | 화면 배경 | 라이트/다크 모드에 따라 전환된다 |
| **Surface** | 카드·시트 배경 | Background 위에 떠 있는 요소의 배경 |
| **On-Primary** | Primary 위 텍스트 | Primary 배경 위에서 가독성을 확보한다 |
| **On-Surface** | Surface 위 텍스트 | Surface 배경 위의 기본 텍스트 색상 |
| **Separator** | 구분선 | 미세한 구분선에 사용한다 |

### 5.2 대비 기준

WCAG 대비 기준을 준수한다.

| 등급 | 대비 비율 | 적용 대상 |
|------|----------|----------|
| **AA (필수)** | 4.5 : 1 이상 | 일반 텍스트 (본문, 버튼 레이블) |
| **AA Large** | 3 : 1 이상 | 18pt 이상 굵은 텍스트, 24pt 이상 일반 텍스트 |
| **AAA (권장)** | 7 : 1 이상 | 핵심 정보, 장시간 읽기 콘텐츠 |
| **비텍스트 요소** | 3 : 1 이상 | 아이콘, 입력 필드 테두리, 포커스 인디케이터 |

### 5.3 다크 모드 설계

| 규칙 | 설명 |
|------|------|
| **시맨틱 컬러 전환** | 동일한 토큰을 사용하되, 실제 색상값을 라이트/다크에 맞게 전환한다 |
| **순수 검정 배경 지양** | `#000000` 대신 어두운 회색(`#1C1C1E` 등)을 배경으로 사용한다 |
| **채도 조정** | 어두운 배경에서 높은 채도는 눈의 피로를 유발한다. 채도를 낮춘다 |
| **엘리베이션 표현** | 그림자 대신 배경색 밝기로 계층을 구분한다 |
| **대비 재검증** | 다크 모드 전환 시 모든 텍스트의 대비 비율을 재확인한다 |

### 5.4 색상 사용 금지 규칙

**색상만으로 정보를 전달하지 않는다.** 색각 이상(색맹·색약) 사용자를 고려하여 반드시 보조 수단을 함께 사용한다.

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 에러 필드에 빨간 테두리 + 에러 아이콘 + 텍스트 메시지 | 에러 필드에 빨간 테두리만 적용 |
| 성공 상태에 녹색 배지 + "완료" 텍스트 | 녹색 원만으로 성공 표현 |
| 차트에서 색상 + 패턴 + 범례 레이블 | 차트에서 색상만으로 구분 |

## 6. 아이콘 & 이미지

### 6.1 아이콘 원칙

| 원칙 | 설명 |
|------|------|
| **간결성** | 불필요한 디테일을 제거하고 핵심 형태만 표현한다 |
| **일관성** | 동일 선 두께, 동일 시각적 무게, 동일 스타일을 유지한다 |
| **정렬** | 아이콘 세트 전체가 광학적으로 균등한 크기와 무게를 갖는다 |
| **인식성** | 의미가 즉시 전달되는 보편적 형태를 사용한다 |

### 6.2 아이콘 사용 규칙

| 규칙 | 설명 |
|------|------|
| 의미 있는 레이블 병기 | 아이콘만 단독 사용하면 의미 전달이 불확실하다. 텍스트 레이블을 함께 표시한다 |
| 상태 변화 반영 | 토글 아이콘은 현재 상태를 반영한다 (예: 음소거 상태에서 음소거 아이콘) |
| 장식 아이콘 최소화 | 기능 없이 장식 목적으로만 사용하는 아이콘을 남발하지 않는다 |

### 6.3 앱 아이콘 규칙

| 규칙 | 설명 |
|------|------|
| **단순한 형태** | 멀리서도 인식 가능한 단일 초점 요소를 사용한다 |
| **텍스트 배제** | 앱 이름을 아이콘 안에 넣지 않는다 |
| **배경 채움** | 아이콘 배경을 채워 투명 영역을 만들지 않는다 |

### 6.4 이미지 접근성

| 유형 | 처리 방법 |
|------|----------|
| **정보 이미지** | 이미지의 내용을 설명하는 대체 텍스트(alt text)를 제공한다 |
| **장식 이미지** | 대체 텍스트를 빈 값으로 설정하여 스크린 리더가 무시하도록 한다 |
| **기능 이미지** | 이미지가 수행하는 기능을 설명한다 (예: "검색", "닫기") |
| **복잡한 이미지** | 차트·다이어그램은 별도 텍스트 설명 또는 데이터 테이블을 제공한다 |

## 7. 컴포넌트

### 7.1 버튼

버튼은 시각적 계층 구조를 통해 중요도를 전달한다.

| 계층 | 스타일 | 용도 |
|------|--------|------|
| **Primary** | 채워진 배경 + 대비 텍스트 | 화면의 주요 액션 (1개) |
| **Secondary** | 테두리 또는 톤 배경 | 보조 액션 |
| **Tertiary** | 텍스트만 (밑줄 없음) | 부가 액션, 취소 |

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 화면당 Primary 버튼 1개 | 화면에 Primary 버튼 3개 병렬 배치 |
| 버튼 레이블에 동사 사용 ("저장", "삭제") | 버튼 레이블에 모호한 텍스트 ("확인", "예") |
| 비활성 상태에서 시각적으로 구분 | 비활성 버튼을 완전히 숨김 |
| 파괴적 액션에 Destructive 스타일 적용 | 삭제 버튼에 Primary 스타일 적용 |

### 7.2 네비게이션 바

| 규칙 | 설명 |
|------|------|
| **제목 표시** | 현재 화면의 맥락을 전달하는 제목을 표시한다 |
| **뒤로 가기** | 이전 화면의 제목 또는 "뒤로"를 레이블로 사용한다 |
| **액션 버튼** | 우측에 배치하며 최대 2개로 제한한다 |
| **커스텀 지양** | 표준 네비게이션 바 형태와 동작을 유지한다 |

### 7.3 탭 바

| 규칙 | 설명                                                   |
|------|------------------------------------------------------|
| **탭 개수** | 제한을 두지 않는다. 단, 화면 너비를 초과할 경우 사이드 버튼을 통해 이동할 수 있어야 한다 |
| **아이콘 + 레이블** | 아이콘과 텍스트 레이블을 함께 표시한다                                |
| **현재 탭 강조** | 선택된 탭을 시각적으로 명확히 구분한다                                |
| **고정 위치** | 탭 바는 항상 동일한 위치에 표시한다. 스크롤에 따라 숨기지 않는다                |
| **일관된 순서** | 탭 순서를 화면에 따라 변경하지 않는다                                |

### 7.4 리스트 / 테이블

| 규칙 | 설명 |
|------|------|
| **Disclosure Indicator** | 하위 화면으로 이동하는 항목에 표시한다 |
| **Swipe Action** | 자주 사용하는 동작(삭제, 아카이브 등)을 스와이프로 제공한다 |
| **섹션 헤더** | 관련 항목을 그룹화하여 탐색성을 높인다 |
| **빈 상태** | 데이터가 없을 때 안내 메시지와 액션을 제공한다. 빈 화면만 표시하지 않는다 |

### 7.5 모달 / 시트

| 사용 시점 | 부적절한 사용 |
|----------|-------------|
| 독립적이고 완결된 작업 (생성, 편집) | 단순 정보 표시 (알림으로 대체) |
| 사용자의 명시적 확인이 필요한 경우 | 탐색 흐름 중간에 강제 삽입 |
| 현재 맥락에서 잠시 벗어나는 작업 | 연속 모달 (모달 위에 모달) |

| 규칙           | 설명                                                       |
|--------------|----------------------------------------------------------|
| **닫기 수단 명확** | 배경 탭을 통한 닫기를 금지한다. 반드시 명시적인 버튼을 통해서만 닫을 수 있어야 한다 |
| **연속 모달 허용** | 모달 위에 모달 호출을 허용한다. 단 최상위 모달만 노출하며 하위 모달은 숨김처리한다          |
| **상태 보존**    | 상위 모달이 열려도 하위 모달의 입력 내용과 스크롤 위치 등 상태는 그대로 유지되어야 한다       |
| **상태 연동**    | 상위 모달의 작업 결과가 하위 모달의 데이터에 즉각 반영될 수 있어야 한다             |
| **순차적 복구**    | 최상위 모달이 닫히면 직전 단계의 모달이 hidden 해제되며 이전 상태 그대로 사용자에게 다시 표시된다 |



### 7.6 폼 / 입력

| 요소 | 규칙 |
|------|------|
| **레이블** | 모든 입력 필드에 고정 레이블을 표시한다. 플레이스홀더로 대체하지 않는다 |
| **플레이스홀더** | 입력 형식 힌트로만 사용한다 (예: "YYYY-MM-DD"). 필드 설명으로 사용하지 않는다 |
| **에러 상태** | 필드 하단에 에러 메시지를 표시한다. 필드 테두리 색상도 변경한다 |
| **성공 상태** | 실시간 검증 통과 시 시각적 확인을 제공한다 (선택적) |
| **필수 표시** | 필수 필드에 시각적 표시(`*` 등)를 제공한다 |
| **키보드 타입** | 이메일 필드에는 이메일 키보드, 전화번호에는 숫자 키보드를 표시한다 |

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 레이블: "이메일", 플레이스홀더: "user@example.com" | 플레이스홀더만: "이메일을 입력하세요" |
| 입력 직후 인라인 에러 + 해결 방법 안내 | "유효하지 않은 값입니다"만 표시 |

**유효성 검사 시점**

| 시점 | 용도 | 주의사항 |
|------|------|---------|
| **입력 중 (onChange)** | 형식 검사 (이메일·URL 형식) | 입력 시작 즉시 에러를 표시하지 않는다 |
| **포커스 이탈 (onBlur)** | 필수 입력, 길이 검사 | 가장 일반적인 검사 시점이다 |
| **제출 (onSubmit)** | 전체 폼 최종 검증 | 서버 에러, 중복 검사는 이 시점에 처리한다 |

### 7.7 데이터 테이블

**컬럼 설계**

| 규칙 | 설명 |
|------|------|
| 컬럼 너비 | 콘텐츠 유형에 맞게 설정한다. 짧은 코드는 좁게, 설명 텍스트는 넓게 |
| 숫자·날짜 우측 정렬 | 3.5 정렬 원칙을 따른다 |
| 식별자 컬럼 고정 | 가로 스크롤 시 ID·이름 컬럼을 좌측에 고정한다 |
| 컬럼 수 제한 | 한 화면에 표시할 컬럼은 7±2개를 기준으로 한다 |

**정렬**

| 규칙 | 설명                                    |
|------|---------------------------------------|
| 정렬 가능 컬럼 표시 | 정렬 아이콘(↑↓)으로 정렬 가능 여부를 나타낸다           |
| 현재 정렬 상태 강조 | 정렬된 컬럼 헤더를 시각적으로 구분한다 (🔺,🔻)         |
| 기본 정렬 명시 | 화면 진입 시 어떤 기준으로 정렬됐는지 사용자가 알 수 있어야 한다 |

**행 선택 & 대량 액션**

| 규칙 | 설명 |
|------|------|
| 체크박스 + 전체 선택 | 헤더 체크박스로 현재 페이지 전체를 선택·해제한다 |
| 선택 수 표시 | "3개 선택됨" 형식으로 상단에 표시한다 |
| 파괴적 대량 액션 확인 | 대량 삭제 등은 반드시 확인 대화상자를 거친다 |

**페이지네이션**

**페이지 번호** 방식을 사용한다. 더 보기와 무한 스크롤은 사용하지 않는다.

| 규칙 | 설명                                          |
|------|---------------------------------------------|
| 총 건수 표시 | 전체 결과 수를 항상 표시한다                            |
| 현재 위치 표시 | 현재 페이지 번호를 시각적으로 강조한다                       |
| 페이지당 건수 선택 | 10/20/50/100/500 중 선택할 수 있도록 제공한다 (기본값은 10) |
| 첫/마지막 페이지 이동 | 페이지 수가 많을 경우 첫·마지막 페이지로 빠르게 이동하는 수단을 제공한다   |

### 7.8 알림 (Alert / Confirm)

Alert과 Confirm 두 가지 다이얼로그만 사용한다. Toast·인라인 배너는 사용하지 않는다.

**Alert — CRUD 결과 및 서버 에러 알림**

| 상황 | 메시지 구성 |
|------|-----------|
| 생성·수정·삭제 성공 | 완료된 작업을 구체적으로 명시한다 ("사용자가 저장되었습니다.") |
| 유효성 검사 실패 | 실패한 필드와 이유를 명시한다 |
| 서버 에러 | 에러 원인 + 에러 코드 또는 참조 ID를 함께 표시한다 |

| 규칙 | 설명 |
|------|------|
| 구체적인 메시지 | "오류가 발생했습니다" 같은 모호한 메시지는 사용하지 않는다 |
| 확인 버튼 1개 | Alert은 확인 버튼만 제공한다 |
| 아이콘 + 텍스트 병기 | 성공·에러 유형을 색상만으로 구분하지 않는다 |

**Confirm — 파괴적 동작 전 사용자 확인**

| 규칙 | 설명 |
|------|------|
| 사용 시점 | 삭제·초기화 등 되돌릴 수 없는 동작 전에 반드시 표시한다 |
| 버튼 레이블 | "확인/취소" 대신 동작을 명시한다 ("삭제/취소", "초기화/취소") |
| 파괴적 버튼 스타일 | 확정 버튼에 Destructive 스타일을 적용한다 |
| 기본 포커스 | 기본 포커스는 취소 버튼에 두어 실수를 방지한다 |

### 7.9 시스템 상태 패턴

데이터를 표시하는 모든 뷰는 5가지 상태를 처리해야 한다.

| 상태 | 조건 | 필수 요소 |
|------|------|----------|
| **Loading** | 데이터 요청 중 | 스켈레톤 또는 스피너 + 접근성 레이블 |
| **Empty** | 데이터 0건 | 아이콘 + 안내 문구 |
| **Error** | 요청 실패 | 에러 설명 + 재시도 버튼 |
| **Success** | 정상 표시 | 데이터 + 메타 정보 (총 건수, 마지막 업데이트 등) |

**빈 상태 (Empty State)**

| 규칙 | 설명 |
|------|------|
| 구체적인 안내 | "데이터가 없습니다"보다 "아직 등록된 사용자가 없습니다"처럼 맥락에 맞게 작성한다 |
| 검색 결과 구분 | 검색 결과 없음은 일반 빈 상태와 구분하고, 검색어 초기화 버튼을 제공한다 |

**에러 상태**

| 규칙 | 설명 |
|------|------|
| 재시도 수단 | 네트워크 오류 시 항상 "다시 시도" 버튼을 제공한다 |
| 에러 코드 표시 | 디버깅을 위해 에러 코드나 참조 ID를 접이식 또는 작은 텍스트로 표시한다 |

## 8. 네비게이션

### 8.1 3가지 패턴

| 패턴 | 구조 | 용도 |
|------|------|------|
| **계층형 (Push)** | A → B → C (스택) | 콘텐츠 계층 탐색 (설정, 카테고리) |
| **플랫 (Tab)** | A ↔ B ↔ C (병렬) | 독립적인 주요 영역 간 전환 |
| **모달** | 기존 흐름 위에 오버레이 | 독립 작업, 확인, 생성 |

### 8.2 패턴 선택 기준

| 조건 | 권장 패턴 |
|------|----------|
| 콘텐츠 간 상하 관계가 있다 | 계층형 |
| 최상위 영역이 3~5개이고 동등하다 | 플랫 (탭) |
| 현재 흐름을 유지하면서 부가 작업을 수행한다 | 모달 |
| 사용자가 빈번하게 영역 간 전환한다 | 플랫 (탭) |
| 작업 완료 후 원래 위치로 복귀해야 한다 | 모달 |

### 8.3 네비게이션 규칙

| 규칙 | 설명 |
|------|------|
| **현재 위치 표시** | 사용자가 앱 내 어디에 있는지 항상 알 수 있도록 한다 |
| **경로 예측 가능** | 동일한 동작이 항상 동일한 결과로 이어진다 |
| **뒤로 가기 일관성** | 뒤로 가기는 항상 이전 화면으로 이동한다 |

### 8.4 딥 링크와 상태 복원

| 규칙 | 설명 |
|------|------|
| **딥 링크 지원** | 앱 내 주요 화면에 직접 진입할 수 있는 경로를 제공한다 |
| **네비게이션 스택 복원** | 딥 링크로 진입해도 뒤로 가기가 자연스럽게 동작한다 |
| **상태 복원** | 앱 재시작 시 마지막 화면과 상태를 복원한다 |

## 9. 인터랙션 & 피드백

### 9.1 표준 인터랙션 매핑

| 인터랙션 | 기본 동작 | 주의사항 |
|--------|----------|---------|
| **클릭** | 선택, 활성화 | 가장 기본적인 상호작용. 모든 인터랙티브 요소에 적용한다 |
| **호버** | 시각적 피드백 | 버튼·행 위에서 상태 변화를 표시한다. 필수 기능을 호버에만 배치하지 않는다 |
| **드래그 앤 드롭** | 이동, 재정렬 | 드래그 가능 요소에 시각적 어포던스를 제공한다 |
| **키보드 Tab** | 포커스 이동 | 모든 인터랙티브 요소가 Tab으로 접근 가능해야 한다 |
| **Enter / Space** | 활성화 | 버튼·링크를 키보드로 실행할 수 있어야 한다 |

### 9.2 애니메이션 원칙

| 원칙 | 설명 |
|------|------|
| **목적이 있는 모션** | 장식 목적의 애니메이션을 사용하지 않는다. 방향, 관계, 결과를 전달한다 |
| **300ms 이하** | 인터페이스 전환 애니메이션은 **200~300ms** 이내로 완료한다 |
| **자연스러운 감속** | ease-out 또는 spring 커브를 사용한다. 선형 모션은 부자연스럽다 |
| **연속성** | 화면 전환 시 공유 요소의 연속적 움직임으로 맥락을 유지한다 |
| **모션 감소 지원** | `prefers-reduced-motion` 설정 시 애니메이션을 최소화하거나 제거한다 |

### 9.3 피드백 유형

| 유형 | 적용 시점 | 예시 |
|------|----------|------|
| **시각** | 모든 인터랙션 | 버튼 하이라이트, 상태 색상 변화, 체크마크 |
| **청각** | 주요 이벤트 | 메시지 전송 완료, 에러 발생 |
| **햅틱** | 중요한 순간 | 토글 전환, 선택 확인, 에러 |

- 시각 피드백은 **필수**다. 청각·햅틱은 보조적으로 사용하며, 과도한 피드백은 지양한다.

### 9.4 로딩 상태 패턴

| 패턴 | 용도 | 기준 |
|------|------|------|
| **스켈레톤** | 콘텐츠 구조가 예측 가능한 경우 | 레이아웃이 고정된 화면 |
| **스피너** | 짧은 대기 (2초 이내) | 버튼 내부, 소형 영역 |
| **프로그레스 바** | 진행률을 알 수 있는 경우 | 파일 업로드, 다운로드 |
| **풀 스크린 로딩** | 화면 전체를 표시할 수 없는 경우 | 초기 로딩 |

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 리스트 화면에 스켈레톤 UI 표시 | 빈 화면에 중앙 스피너만 표시 |
| 2초 이상 소요 시 진행 상태 표시 | 로딩 표시 없이 화면 멈춤 |

## 10. 접근성

### 10.1 기본 원칙

접근성은 선택이 아니라 **필수**다. 모든 사용자가 동등하게 앱을 사용할 수 있도록 설계한다.

| 원칙 | 설명 |
|------|------|
| **인식 가능** | 모든 정보를 시각·청각·촉각 중 하나 이상으로 인식할 수 있다 |
| **조작 가능** | 모든 기능을 다양한 입력 방식(터치, 키보드, 스위치 등)으로 사용할 수 있다 |
| **이해 가능** | 콘텐츠와 동작이 예측 가능하고 일관된다 |
| **견고함** | 보조 기술(스크린 리더 등)과 호환된다 |

### 10.2 브라우저 텍스트 줌 대응

| 규칙 | 설명 |
|------|------|
| **rem 기반 폰트 크기** | px 고정값 대신 rem 단위를 사용한다. 브라우저 기본 폰트 크기 변경에 대응한다 |
| **레이아웃 유연성** | 텍스트 크기가 커져도 잘리거나 겹치지 않도록 한다 |
| **200% 확대 테스트** | 브라우저 텍스트 200% 확대 시 레이아웃이 정상 동작하는지 확인한다 |

### 10.3 스크린 리더 레이블

| 규칙 | 설명 |
|------|------|
| **인터랙티브 요소 레이블** | 버튼, 링크, 입력 필드에 명확한 접근성 레이블을 부여한다 |
| **이미지 대체 텍스트** | 정보 이미지에 내용을 설명하는 텍스트를 추가한다 |
| **장식 요소 숨김** | 의미 없는 장식 요소는 스크린 리더가 무시하도록 처리한다 |
| **읽기 순서 일치** | 화면의 논리적 순서와 스크린 리더의 탐색 순서를 맞춘다 |

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 닫기 버튼: 레이블 "닫기" | 닫기 버튼: 레이블 없음 (아이콘만) |
| 장바구니 버튼: "장바구니, 3개 항목" | 장바구니 버튼: "버튼" |
| 이미지: "산 위의 일출 사진" | 이미지: "image_01.jpg" |

### 10.4 모션 감소

| 규칙 | 설명 |
|------|------|
| **`prefers-reduced-motion` 존중** | 사용자가 모션 감소를 설정하면 애니메이션을 제거하거나 최소화한다 |
| **필수 모션만 유지** | 의미 전달에 필수적인 전환은 유지하되, 즉시 전환(cross-dissolve)으로 대체한다 |
| **자동 재생 금지** | 모션 감소 모드에서 자동 재생 애니메이션·영상을 중지한다 |

### 10.5 색상 대비 (재강조)

모든 텍스트는 **AA 기준(4.5:1)** 이상의 대비를 확보한다. 색상만으로 정보를 전달하지 않는다. 다크 모드 전환 후 대비를 재검증한다 — §5.2절, §5.4절 참고.

### 10.6 포커스 관리

키보드 및 보조 기기 사용자를 위한 포커스 제어 규칙이다.

**포커스 링 (Focus Ring)**

| 규칙 | 설명 |
|------|------|
| 항상 가시적 | `outline: none`으로 포커스 링을 제거하지 않는다 |
| 최소 대비 | 포커스 인디케이터와 배경의 대비 비율은 3:1 이상이어야 한다 |
| 커스텀 허용 | 브라우저 기본 스타일을 커스텀 디자인으로 교체할 수 있으나, 가시성은 반드시 유지한다 |

**모달 포커스 트랩**

| 규칙 | 설명 |
|------|------|
| 포커스 진입 | 모달 열림 시 첫 번째 인터랙티브 요소 또는 모달 제목으로 포커스를 이동한다 |
| 포커스 순환 | Tab이 모달 내부 요소 사이에서만 순환하도록 트랩을 적용한다 |
| 포커스 복원 | 모달 닫힘 시 모달을 열었던 트리거 요소로 포커스를 반환한다 |

**탭 순서**

| 규칙 | 설명 |
|------|------|
| 논리적 순서 | 시각적 레이아웃 순서와 DOM Tab 순서를 일치시킨다 |
| `tabindex` 최소화 | 양수 `tabindex` 값(1 이상)을 사용하지 않는다. `tabindex="0"` 또는 `-1`만 허용한다 |
| 숨김 요소 제외 | `display: none`·`visibility: hidden` 요소는 Tab 순서에서 제외되어야 한다 |

| 올바른 예 | 잘못된 예 |
|-----------|-----------|
| 모달 닫기 후 "편집" 버튼으로 포커스 복귀 | 모달 닫기 후 포커스가 페이지 최상단으로 이동 |
| Tab으로 폼 전체를 순서대로 탐색 가능 | 시각적 순서와 Tab 순서가 불일치 |

## 11. 금지 사항

### 11.1 UI / UX

| 금지 항목 | 이유 | 대안                                  |
|----------|------|-------------------------------------|
| 색상만으로 상태 표시 | 색각 이상 사용자가 정보를 인식할 수 없다 | 색상 + 아이콘(선택) + 텍스트를 함께 사용한다         |
| 플레이스홀더를 레이블로 사용 | 입력 시작 시 레이블이 사라져 맥락을 잃는다 | 고정 레이블 + 형식 힌트용 플레이스홀더를 분리한다        |
| 표준 아이콘의 의미 변경 | 사용자가 학습한 매핑을 깨뜨려 혼란을 유발한다 | 표준 의미를 유지하거나, 새로운 아이콘을 만들어 의미를 정의한다 |
| 가로 스크롤 텍스트 (마키) | 읽기 속도를 통제할 수 없고 접근성을 저해한다 | 텍스트를 줄이거나 말줄임표(`…`)로 처리한다           |
| 확인 없는 파괴적 동작 | 실수로 데이터를 잃을 위험이 있다 | 삭제·초기화 전 Confirm 다이얼로그를 제공한다        |

### 11.2 HTML / 템플릿

| 금지 항목 | 이유 | 대안 |
|----------|------|------|
| Inline `style=""` 작성 | 스타일이 분산되어 유지보수가 어려워진다 | Tailwind 유틸리티 클래스 또는 `app.css`를 사용한다 |
| `th:insert` 사용 | 불필요한 wrapper 태그가 생성된다 | `th:replace`를 사용한다 |
| `th:object` / `th:field` / `th:with` 사용 | JS 컴포넌트 방식과 충돌한다 | fragment 파라미터 또는 JS 변수로 대체한다 |
| Page fragment에 완전한 HTML 문서 구조 작성 | `#dynamic-content` 주입 시 DOM 구조가 깨진다 | `<div class="content-inner">`로 시작하는 fragment 형식으로 작성한다 |
| 공통 fragment에 도메인 고유 ID 하드코딩 | 다른 도메인에서 ID 충돌이 발생한다 | 도메인 스크립트에서 JS로 동적 처리한다 |
| Raw hex 색상값 HTML에 직접 작성 | 색상 토큰 일관성이 깨진다 | `tailwind.config.js`의 커스텀 색상 토큰을 사용한다 |
| CDN 중복 import | 리소스 낭비 및 충돌 위험이 있다 | jQuery 등 공통 라이브러리는 `home.html`에서만 로드한다 |
| `th:utext` 무분별 사용 | XSS 취약점이 발생한다 | HTML 마크업이 포함된 데이터에만 허용하고, 기본은 `th:text`를 사용한다 |

```html
<!-- ❌ Inline style -->
<div style="background: #3b2e5a; padding: 8px;">

    <!-- ✅ Tailwind 토큰 클래스 -->
    <div class="bg-sidebar p-2">


        <!-- ❌ th:insert — wrapper div 추가됨 -->
        <div th:insert="~{fragments/sidebar :: sidebar}"></div>

        <!-- ✅ th:replace — wrapper 없이 대체 -->
        <div th:replace="~{fragments/sidebar :: sidebar}"></div>


        <!-- ❌ Page fragment에 완전한 HTML 문서 구조 -->
        <!DOCTYPE html>
        <html><body> ... </body></html>

        <!-- ✅ content-inner div로 시작하는 fragment -->
        <div class="content-inner">
            <div id="searchContainer"></div>
            <div id="tableContainer"></div>
        </div>


        <!-- ❌ th:utext 기본 사용 — XSS 위험 -->
        <td th:utext="${item.description}"></td>

        <!-- ✅ th:text 기본, HTML 마크업 포함 데이터에만 th:utext 허용 -->
        <td th:text="${item.description} ?: '(없음)'"></td>


        <!-- ❌ Raw hex 직접 작성 -->
        <div class="bg-[#3b2e5a]">
            <div style="color: #4a7cc9">

                <!-- ✅ 커스텀 색상 토큰 사용 -->
                <div class="bg-sidebar">
                    <div class="text-primary">
```

### 11.3 JavaScript

| 금지 항목 | 이유 | 대안 |
|----------|------|------|
| `var` 사용 | 호이스팅으로 예측하기 어려운 버그가 발생한다 | `let` / `const`를 사용한다 |
| 암묵적 전역 변수 선언 | 탭 간 상태 오염이 발생한다 | `window.{Domain}Page` 네임스페이스에 명시적으로 등록한다 |
| `==` 비교 연산자 | 타입 강제 변환으로 의도치 않은 결과가 발생한다 | `===`를 사용한다 |
| `fetch()` 사용 | CSRF 처리·에러 핸들링 방식이 달라 일관성이 깨진다 | `$.ajax()`로 통일한다 |
| `addEventListener` 직접 사용 | 탭 재진입 시 이벤트가 중복 등록된다 | jQuery `.off().on()` 패턴을 사용한다 |
| 크레덴셜 하드코딩 (Playwright) | 소스 코드에 인증 정보가 노출된다 | 환경변수(`process.env.*`) 또는 `.env` 파일을 사용한다 |

```javascript
// ❌ var, 암묵적 전역, ==, fetch, addEventListener
var count = 0;
const MyPage = { init() {} };           // 암묵적 전역
if (status == '1') { ... }              // == 비교
const res = await fetch('/api/items');  // fetch 금지
document.getElementById('btn').addEventListener('click', load); // 중복 등록 위험

// ✅ const/let, window 네임스페이스, ===, $.ajax, .off().on()
const count = 0;
window.MyPage = { init() {} };          // window 명시
if (status === '1') { ... }             // === 비교
$.ajax({ url: '/api/items', method: 'GET', success: (res) => { ... } });
$('#btn').off('click').on('click', load); // 탭 재진입 안전


// ❌ 크레덴셜 하드코딩 (Playwright)
const LOGIN_ID = 'admin';
const LOGIN_PW = 'password123!';

// ✅ 환경변수 사용
const LOGIN_ID = process.env.CI_TEST_USER_ID;
const LOGIN_PW = process.env.CI_TEST_PASSWORD;
```

## 12. 프론트엔드 도구

### 12.1 Tailwind CSS

**버전:** 3.4.x | **설정 파일:** `tailwind.config.js`

유틸리티 클래스 기반 스타일링 시스템이다. 모든 스타일은 Tailwind 유틸리티 클래스 또는 `app.css`(`@apply` 기반)로 작성한다. Inline `style=""` 및 Raw hex 색상값 직접 작성은 금지한다 — §11.2절 참고.

**`tailwind.config.js`:**

§5.1절 시맨틱 컬러 토큰을 Tailwind 커스텀 색상으로 등록하여, 하드코딩된 색상값 없이 의미 있는 클래스명을 사용한다.

```js
// tailwind.config.js
module.exports = {
  content: [
    './src/main/resources/templates/**/*.html',
    './src/main/resources/static/js/**/*.js',
  ],
  theme: {
    extend: {
      colors: {
        // §5.1 시맨틱 컬러 토큰
        primary:     '#4a7cc9',   // 주요 액션, 브랜드 강조
        secondary:   '#6b7280',   // 보조 액션
        destructive: '#ef4444',   // 삭제, 경고
        success:     '#22c55e',   // 성공 상태
        warning:     '#f59e0b',   // 주의 상태
        // 레이아웃 전용
        sidebar:     '#3b2e5a',   // 사이드바 배경
        topbar:      '#1e293b',   // 상단 바 배경
        surface:     '#ffffff',   // 카드·시트 배경
        separator:   '#e5e7eb',   // 구분선
      },
      // §3.1 8pt 그리드 — Tailwind 기본 spacing(4px 단위)과 일치
      // p-2=8px, p-4=16px, p-6=24px, p-8=32px
    },
  },
}
```

**빌드 방식:**

전 환경에서 PostCSS 빌드만 사용한다. CDN 방식은 Security.md §8.1의 CSP 정책(`style-src 'self'`)에 의해 차단되므로 로컬 개발 환경에서도 사용하지 않는다.

| 방식 | 환경 | 명령어 |
|------|------|--------|
| PostCSS 빌드 | 전 환경 (로컬·CI·운영) | 로컬: `npm run css:watch` / CI·운영: `npm run css:build` |

> 로컬 개발 시 `npm run css:watch`를 백그라운드로 실행하면 파일 변경 시 자동 재빌드된다.

**`package.json` 프론트엔드 스크립트:**

```json
{
  "scripts": {
    "css:build": "tailwindcss -i ./src/main/resources/static/css/input.css -o ./src/main/resources/static/css/app.css --minify",
    "css:watch": "tailwindcss -i ./src/main/resources/static/css/input.css -o ./src/main/resources/static/css/app.css --watch",
    "lint":      "eslint src/main/resources/static/js && htmlhint 'src/main/resources/templates/**/*.html'",
    "lint:fix":  "eslint src/main/resources/static/js --fix"
  },
  "devDependencies": {
    "globals": "^15.0.0"
  }
}
```

**Thymeleaf 연동:**

빌드된 CSS 파일(`app.css`)을 `src/main/resources/static/css/`에 출력하며, 공통 레이아웃에서 한 번만 로드한다.

```html
<!-- fragments/layout.html -->
<link rel="stylesheet" th:href="@{/css/app.css}">
```

**Tailwind 클래스 사용 패턴:**

| 목적 | 예시 |
|------|------|
| 시맨틱 컬러 | `class="bg-primary text-white"` |
| 간격 (8pt 기반) | `class="p-4 mt-2 gap-4"` |
| 레이아웃 | `class="flex items-center justify-between"` |
| 반응형 | `class="hidden lg:block"` |
| 비활성 상태 | `class="opacity-50 cursor-not-allowed"` |

### 12.2 jQuery

**버전:** 3.7.x

DOM 조작, AJAX 통신, 이벤트 처리에 사용한다. `fetch()` 대신 `$.ajax()`를 사용하여 CSRF 토큰 처리와 에러 핸들링을 일관되게 적용한다. 이벤트는 탭 재진입 안전성을 위해 `.off().on()` 패턴을 사용한다 — §11.3절 참고.

공통 라이브러리는 `layout.html`에서 **한 번만** 로드한다. 도메인별 스크립트에서 중복 import는 금지한다 — §11.2절 CDN 중복 import 참고.

```html
<!-- fragments/layout.html (단일 로드) -->
<script th:src="@{/js/vendor/jquery.min.js}"></script>
```

### 12.3 ESLint

**버전:** 9.x | **설정 파일:** `eslint.config.js`

JavaScript 코드 품질을 강제한다. §11.3절 금지 패턴을 자동으로 검출하고 CI 빌드를 차단한다 — §11.4절 강제 가능 항목 참고.

`globals` 패키지로 브라우저 및 jQuery 전역 변수를 명시적으로 선언한다. 선언하지 않으면 `no-implicit-globals` 규칙이 `$`, `window`, `document` 등 정상 코드에서 오류를 발생시킨다.

```bash
npm install -D globals
```

```js
// eslint.config.js
import globals from 'globals';

export default [{
  files: ['src/main/resources/static/js/**/*.js'],
  languageOptions: {
    globals: {
      ...globals.browser,   // window, document 등 브라우저 전역
      ...globals.jquery,    // $ (jQuery) 전역
    },
  },
  rules: {
    'no-var':                'error',
    'no-implicit-globals':   'error',
    'eqeqeq':               'error',
    'prefer-const':          'warn',
    'no-restricted-globals': ['error', { name: 'fetch', message: 'Use $.ajax() instead.' }],
    'no-restricted-syntax':  ['warn',
      { selector: "CallExpression[callee.property.name='addEventListener']",
        message: 'Use jQuery .off().on() instead.' }
    ],
  },
}];
```
